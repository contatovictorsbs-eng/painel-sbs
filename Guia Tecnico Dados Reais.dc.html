Como ligar os dados reais
da Plataforma SBS

Este guia mostra, módulo a módulo, de onde virá cada informação depois que o painel for publicado — quais fontes usar, quais funções de servidor criar e quais chaves configurar. É o roteiro para entregar ao desenvolvedor que fará a virada de "dados de demonstração" para "dados reais".

Ponto de partida. Hoje o painel funciona 100% no navegador com dados de exemplo. A virada para dados reais não muda o visual: cada tela passa a ler de uma função de servidor (serverless) em vez da base local. O front-end já está pronto para isso — parte das funções (coletor-concorrentes, preco-concorrente) já existe no seu projeto.

1. Como funciona — as três camadas

1

Painel (navegador)

O que já está pronto: 4 painéis, App de Eventos, robôs, clima. Ele apenas lê e mostra — não guarda segredo nenhum. É publicado como site estático.

2

Funções de servidor Cloudflare Pages Functions — grátis

Pequenos scripts que rodam no servidor, guardam as chaves de API com segurança e buscam os dados nas fontes externas. O painel chama estas funções por um endereço /api/nome.

3

Fontes de dados

APIs públicas do agro (CEPEA, CONAB, IBGE), clima (Open-Meteo/INMET), câmbio, e o seu CRM/ERP (pedidos, metas, vendedores). Cada função conversa com uma fonte.

4. Publicar e ativar — passo a passo

1

Publicar o painel. Conecte o repositório no Cloudflare Pages (Workers & Pages → Create → Pages → Connect to Git). O site fica no ar com HTTPS; cada branch/PR ganha uma URL de preview e a main vira produção.

2

Ativar as funções. O roteador functions/api/[[path]].js já publica cada arquivo de server/ em /api/nome. Ligue a flag nodejs_compat em Settings → Functions → Compatibility flags (Production e Preview).

3

Cadastrar as chaves. Em Settings → Environment variables, adicione AV_KEY, AI_KEY, ERP_TOKEN, SUPABASE_URL, SUPABASE_SERVICE_KEY etc. Assim as chaves ficam no servidor, fora do alcance do navegador.

4

Ligar módulo a módulo. Recomendo a ordem: Clima (mais fácil, sem chave) → Câmbio/Cotações → Concorrentes → CRM/ERP (Insights e Regiões) → Panorama. Cada um pode entrar no ar separadamente.

5

Agendar os robôs. Configure Scheduled Functions para coletor-concorrentes e alertas rodarem sozinhos (ex.: de 6 em 6 horas), gravando os achados na base.

5. A ponte com o CRM/ERP

É a integração mais importante — dela saem Insights de vendas, Regiões, metas e os vendedores. Duas formas:

  • API do seu ERP — a função vendedores/vendas consulta o ERP e devolve ao painel. Melhor para dados sempre atualizados.
  • Exportação periódica — se o ERP não tiver API, um arquivo (CSV/planilha) é importado de tempos em tempos. Mais simples de começar.

Sentido duplo. As vendas e cadastros feitos no App de Eventos precisam gravar de volta na sua base — não só ler. Defina com o time de TI qual sistema é a "fonte da verdade" desses registros de campo.

6. Insights automáticos por IA opcional

Os textos de resumo (panoramas, leitura de mercado) podem ser gerados por IA. O painel envia os números para a função ia-resumo, que chama o modelo (Gemini/Groq) com a chave AI_KEY e devolve o texto. Os dados continuam sendo os reais — a IA só redige a leitura. Comece sem isso; ative quando os dados reais já estiverem fluindo.

7. Segurança e LGPD

  • Chaves só no servidor. Nenhuma credencial no navegador — sempre em variáveis de ambiente.
  • Dados de vendedores e clientes (CNPJ, contatos) são pessoais: acesso por perfil, e uma política de privacidade informando o uso.
  • Login real por perfil (Marketing, Gerente, CEO, Inteligência, Vendedor) antes de publicar com dados reais.
  • Registro de acesso (log) para auditoria de quem vê e edita o quê.
